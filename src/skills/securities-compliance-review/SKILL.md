# 증권 투자권유·광고 준법 심의

## 언제 쓰는가
사용자가 증권업 대고객 콘텐츠(투자권유문자, 상품소개서, SNS광고, 리서치·투자자문
코멘트, 신규상품 안내 이메일)를 배포하기 전 준법 심의를 요청할 때 이 스킬을
사용한다. 콘텐츠 타입이 불명확하면 사용자에게 먼저 확인한다.

## 이 스킬의 출발점
이 스킬은 새로 설계한 것이 아니라, JB금융그룹 Fin AI Challenge(지정주제:
Compliance AI)에 제출했던 개인 프로젝트 `JB-Compliance-Agent`
(`github.com/BaeYoungHwan/JB_LLM`)의 Retrieval → Analysis → Feedback
3-Agent 구조를 증권업 투자권유·광고 규제에 맞게 재구성한 것이다. 원본은
로컬 Ollama LLM과 ChromaDB를 별도 Agent로 호출했지만, 이 스킬에서는
Codex 자신이 Analysis Agent 역할을 직접 수행하고, 결정론적 외부 조회
(법제처 API, 유사사례 검색)만 스크립트로 분리했다.

## 절차

### 1단계 — 콘텐츠 타입 확인
다음 5개 타입 중 하나로 분류한다: `투자권유문자`, `상품소개서`, `SNS광고`,
`리서치·투자자문 코멘트`, `신규상품 안내 이메일`. 사용자가 명시하지 않았다면
콘텐츠 성격을 보고 가장 가까운 타입을 고르되, 애매하면 먼저 물어본다.

### 2단계 — 법령 키워드 + 규칙 기반 사전 위험도 조회 (스크립트, 결정론적)
```
cd scripts && python content_classifier.py "<콘텐츠 타입>" "<본문>"
```
`law_keywords`(적용 법령 후보)와 `preliminary_risk`(규칙 기반 사전 위험도,
없으면 null)를 JSON으로 반환한다. `preliminary_risk`가 있다면 "원금보장",
"확정수익" 등 증권 광고에서 사실상 예외 없이 문제되는 표현이 감지됐다는
뜻이다 — 이는 4단계 최종 판단의 하한선이다.

### 3단계 — 법제처 Open API 조문 조회 (스크립트, 결정론적)
```
python law_search.py <law_keywords 배열의 각 항목>
```
`LAW_API_KEY` 환경변수가 필요하다 (레포 루트 `.env`, `.env.example` 참고).
반환된 조문 리스트가 이후 분석의 유일한 근거 조문이 된다. **API 키가 없거나
호출이 실패하면 즉시 사용자에게 알리고, 조문 없이 위험도를 추정하지 않는다**
(아래 "정보가 부족한 상황" 참고).

### 4단계 — 유사 과거 사례 조회 (스크립트, 결정론적)
```
python case_store.py query "<본문>" --type "<콘텐츠 타입>" --n 3
```
`data/cases.json`에 저장된 과거 심의 이력 중 콘텐츠 타입이 같고 단어 겹침
유사도가 높은 상위 3건을 반환한다. 초기에는 시드 데이터 8건만 있다.

### 5단계 — 분석 (Codex가 직접 수행, Analysis Agent 역할)
아래 형식으로 심의 결과를 작성한다. **3단계에서 실제로 조회된 조문에
등장하지 않는 법령명은 절대 인용하지 않는다** (환각 방지 — 원본
JB-Compliance-Agent의 검증 Agent가 하던 역할을 Codex 스스로 수행).

```
## 위반 위험도: High / Medium / Low / 없음

## 위반 가능 조항
(3단계에서 조회된 조문에서만 직접 인용. 해당 조문이 없으면 "해당 없음")

## 수정 제안
1. (수정 전) -> (수정 후) 형식으로 1~3개

## 심의 의견
(한 문단으로 종합 의견. 2단계 규칙 기반 사전 위험도보다 낮게 판정했다면,
반드시 그 근거를 조회된 조문을 들어 여기에 명시한다. 근거 없이 사전
판정보다 낮추지 않는다.)
```

판단 기준 세부 규칙(위험도 판정 표, 키워드 확장 규칙)은
`references/심의_기준.md`에 있다.

### 6단계 — 인용 검증 (Codex 스스로 재확인)
작성한 "위반 가능 조항"을 3단계 조회 결과와 다시 대조한다. 조회되지 않은
법령이 하나라도 인용됐다면, 그 부분만 "해당 없음"으로 고치거나 실제
조회된 조문으로 교체해 재작성한다. 이 재확인 없이 결과를 확정하지 않는다.

### 7단계 — 준법 관리자 피드백 저장 (스크립트, 결정론적)
사용자(준법 관리자)가 승인/거절/수정 의견을 주면 저장한다.
```
python case_store.py save --content "<본문>" --result approved|rejected|modified \
  --type "<콘텐츠 타입>" --department "<부서명>" \
  --violated-laws "법령1,법령2" --feedback "<피드백 메모>"
```
피드백 메모가 5자 미만이거나 거절인데 위반 법령이 비어 있으면 스크립트가
자동으로 저장을 거부한다 (`saved: false`). 이때 사용자에게 사유를 더
구체적으로 입력해 달라고 요청한다.

## 정보가 부족하거나 실패하는 상황에서의 동작
| 상황 | 동작 |
| --- | --- |
| `LAW_API_KEY` 미설정 또는 법제처 API 호출 실패 | 심의를 중단하고 "근거 법령 조회 실패 — 수동 검토 필요"로 표시. 위험도를 임의로 추정하지 않는다 |
| 3단계에서 관련 조문이 하나도 검색되지 않음 | "근거 법령 없음, 수동 검토 필요"로 표시하고 위험도를 추정하지 않는다 |
| 5단계 분석에서 조회되지 않은 법령을 인용하려는 경우 | 6단계 검증에서 반드시 걸러내고 재작성한다 |
| 준법 관리자 피드백이 형식적(사유 없는 거절 등) | `case_store.py save`가 저장을 거부하므로 사용자에게 재입력을 요청한다 |
| `data/cases.json`이 비어 있거나 유사 사례가 없음 | "(유사 과거 사례 없음)"으로 표시하고 3단계 조문 근거만으로 분석을 진행한다 |

## 참고 문서
- `references/심의_기준.md` — 콘텐츠 타입 분류, 우선 검색 법령, 키워드 확장
  규칙, 위험도 판정 기준, 검증·피드백 저장 원칙 전체
- `data/cases.json` — 과거 심의 이력 (시드 8건 + 승인/거절/수정 피드백 누적)
