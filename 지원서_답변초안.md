## 1. 무엇을, 누가, 어떤 상황에서 쓰나요?

**한 문장 요약**
카카오페이증권 마케팅/영업/디지털 채널 담당자가 투자권유 문자, 상품소개서,
SNS 광고 등 대고객 콘텐츠를 배포하기 전에, 자본시장법·금융소비자보호법상
부당권유·광고 규제 위반 가능성을 사전 심의해주는 Codex 플러그인입니다.

**누가, 어떤 상황에서, 어디서 막히는가**
카카오페이증권 마케팅팀·영업지원팀·디지털 채널 담당자는 투자권유 문자(MTS/HTS
푸시), 펀드·ELS·해외주식 상품소개서, SNS 광고, 신규상품 안내 이메일을
지속적으로 만들어 고객에게 배포합니다. 이 콘텐츠는 배포 전 준법 심의를 거쳐야
하는데, 수작업 심의는 지연되기 쉽고, "원금보장", "확정수익"처럼 자본시장법상
명백히 문제되는 표현조차 담당자 개인의 규제 지식 수준에 따라 판단이 갈립니다.
일반 LLM에 그대로 맡기면 법령을 그럴듯하게 지어내는 환각 위험과, 아직
공개되지 않은 상품 정보가 외부로 전송되는 위험이 있어 그대로 쓰기 어렵습니다.

## 2. 왜 이 문제를 선택했나요?

이 프로젝트는 완전히 새로 시작한 것이 아니라, JB금융그룹 Fin AI Challenge
(지정주제: Compliance AI)에 제출했던 개인 프로젝트 `JB-Compliance-Agent`를
기반으로 합니다. 그 프로젝트를 만들며 확인한 핵심은, 금융권 대고객 콘텐츠
심의에서는 모델 성능보다 "판단 근거를 어디까지 제한하고 어떻게 검증하는가"가
더 중요하다는 점이었습니다.

카카오페이증권 같은 증권사는 이 문제가 더 날카롭게 나타납니다. 증권사
투자광고는 은행 등 일반 금융보다 규제가 촘촘해서(금융소비자보호법 제21조
부당권유행위 금지, 제22조 광고 관련 준수사항 등), "원금보장"이나
"확정수익" 같은 표현은 예금과 혼동을 유발한다는 이유로 사실상 엄격히
제한됩니다. 그래서 기존 프로젝트의 범용 심의 구조를 그대로 두지 않고, 법령
검색 우선순위와 키워드 규칙을 증권업 투자권유·광고 규제에 맞게 재구성하는
쪽을 선택했습니다. 이미 검증된 파이프라인을 새 도메인에 정확히 맞춰 확장하는
것이 "현업에서 통하는" 결과물을 만드는 데 더 현실적이라고
판단했습니다.

실제로 이 판단을 뒷받침하는 공개 자료를 두 가지 확인했다. 첫째, 금융위원회는
증권회사 직원이 관리고객에게 문자메시지로 상품 정보를 보낼 때 투자광고
규제 대상인지를 수신자 규모가 아니라 "신규 거래 유인 목적" 여부로 판단한다는
법령해석을 냈다(자본시장법 투자광고 규정, 금융투자업규정 제4-12조 근거).
기존 상품 수익률 안내는 규제 대상이 아니지만 신규 상품·서비스 홍보 메시지는
투자광고 규제를 따라야 한다는 것으로, 이 프로젝트가 다루는 "투자권유문자"
콘텐츠 타입이 실제로 개별 판단이 필요한 회색지대라는 점을 확인해줬다.
둘째, 금융위원회·금융감독원이 2026년 4월 발표한 유사투자자문업체 영업실태
점검 결과에 따르면 105개 업체에서 133건의 위법행위가 적발되고 35개
업체에 총 4억7천만 원의 과태료가 부과됐는데(전년 22개 업체·1억4천만 원
대비 약 3.3배), 이 중 '부당 표시·광고' 위반이 16건으로, 실현되지 않은
수익률 제시나 손실보전·이익보장 약속이 주요 적발 사유였다. 2024년 8월
시행된 자본시장법 시행령 개정으로 이런 표현 자체가 명시적으로 금지된
이후에도 적발 건수가 오히려 늘었다는 점은, "원금보장"·"확정수익" 류
표현을 LLM의 문맥적 판단에 맡기지 않고 규칙으로 먼저 걸러야 한다는 이
프로젝트의 설계 방향이 현실 규제 환경과 맞다는 근거가 된다.

금융위원회 2024.8.13 보도자료(유사투자자문업자 불건전영업행위 규율

점검 결과 보도(디지털데일리·파이낸셜뉴스 등 보도, 원출처 금융위원회)]

## 3. 플러그인은 어떻게 작동하나요?

**절차 (원본 JB-Compliance-Agent의 Retrieval → Analysis → Feedback 구조를
Codex 플러그인에 맞게 재구성)**
원본은 Retrieval/Analysis/Feedback을 로컬 Ollama·ChromaDB로 구현한 별도
Agent였지만, 이번 제출물은 Codex 플러그인이라 Codex 자신이 Analysis Agent
역할을 직접 수행하고, 결정론적 외부 조회만 스크립트로 분리했다.
1. 담당자가 배포 예정 콘텐츠와 콘텐츠 타입(투자권유문자/상품소개서/SNS광고/
   리서치·투자자문 코멘트/신규상품 안내 이메일)을 입력한다.
2. `content_classifier.py`가 콘텐츠 타입 + 본문 키워드로 금융소비자보호법
   제21조(부당권유행위 금지)·제22조(광고 관련 준수사항), 금융투자업규정
   검색 키워드와 규칙 기반 사전 위험도(예: "원금보장"→High)를 계산한다.
3. `law_search.py`가 법제처 Open API로 실제 조문을 조회한다.
4. `case_store.py`가 JSON 저장소(`data/cases.json`)에서 유사 과거 심의
   사례 Top-3를 단어 겹침 유사도로 조회한다.
5. Codex가 3단계에서 조회된 조문 범위 안에서만 위반 위험도, 위반 가능
   조항, 수정 제안, 심의 의견을 생성한다. 2단계 사전 위험도보다 낮게
   판정하려면 반드시 조문 근거를 명시해야 한다.
6. Codex 스스로 자신이 인용한 법령이 3단계 조회 결과에 실제로 있는지
   재검증한다(환각 방지 — 원본의 검증 Agent 역할).
7. `case_store.py`가 준법 관리자의 승인/거절/수정 피드백을 품질 필터
   (5자 미만 또는 위반 법령 미기재 시 거부)를 거쳐 저장해 심의 이력으로
   누적한다.

**지식/판단 기준을 어떻게 가져가는가**
법령 우선순위, 키워드 확장 규칙, 위험도 판정 기준을 코드에 하드코딩하지 않고
`src/skills/securities-compliance-review/references/심의_기준.md`에 문서로
분리했다. 특히 "원금보장", "확정수익" 계열 표현은 Codex 판단 이전에
`content_classifier.py`의 규칙으로 먼저 위험도 High를 부여하도록 했는데, 이는
LLM(Codex)이 명백한 위반 표현을 완곡하게 평가할 가능성을 규칙으로 먼저
차단하기 위해서다. 최종 조문 근거는 5~6단계에서 Codex가 다시 확인한다.

**정보가 부족하거나 잘 안 풀리는 상황에서의 동작**
- `LAW_API_KEY` 미설정이거나 API 호출이 실패하면 심의를 중단하고 "근거 법령
  조회 실패, 수동 검토 필요"로 표시하며 위험도를 임의로 추정하지 않는다.
- 관련 조문이 하나도 검색되지 않으면 "근거 법령 없음, 수동 검토 필요"로
  표시한다.
- Codex 출력에 검색되지 않은 법령이 인용되면 6단계 자체 검증에서 걸러내고
  재작성한다.
- 준법 관리자 피드백이 형식적이면(사유 없는 거절 등) `case_store.py`가
  저장을 거부해 저품질 데이터가 심의 이력에 쌓이지 않게 한다.

세부 내용은 zip에 포함된 `README.md`,
`src/skills/securities-compliance-review/SKILL.md`,
`src/skills/securities-compliance-review/references/심의_기준.md`에
동일하게 기술되어 있다.

## 4. AI를 어떻게 썼나요?

**AI에게 맡긴 작업**
- 원본 JB-Compliance-Agent 코드 구조(Retrieval/Analysis/Feedback Agent)를
  분석해 증권업 콘텐츠 타입·법령 우선순위로 재구성하는 초안 작성
- 관련 조문 요약과 키워드 매핑 초안 작성 (초안 단계에서는 "자본시장법
  제49조·제57조"로 적었으나, 이후 법제처 API 실검증에서 두 조항이
  2020년에 삭제되고 금융소비자보호법 제21조·제22조로 이관된 사실을
  발견해 전체 문서를 정정했다 — 아래 "막혔던 지점" 참고)
- README/SKILL.md/심의_기준.md 등 문서 초안 작성

**직접 판단한 부분**
- 원본을 폐기하고 새로 만들지, 재사용하고 도메인만 재구성할지의 결정 —
  이미 검증된 파이프라인이 있는데 하루 만에 새로 만드는 것은 리스크가 크다고
  판단해 재사용 쪽을 선택
- "원금보장", "확정수익" 표현에 대해 LLM 판단보다 먼저 규칙으로 High를
  부여하기로 한 결정 — 증권 광고 규제에서는 이 표현이 사실상 예외 없이
  문제되므로, LLM의 유연한 판단보다 규칙의 확정성이 더 안전하다고 봄
- 서류에 "재사용 프로젝트"라는 사실을 숨기지 않고 명시하기로 한 결정 —
  기존 시스템을 새 도메인에 확장 적용하는 능력 자체가 강점이라고 판단

**막혔던 지점과 해결 과정**
실제로 Codex CLI에 플러그인을 로컬 설치해 스킬을 트리거해보면서 두 가지
이슈를 발견했다.

첫째, `codex exec --sandbox workspace-write`(비대화형 자동 승인 모드)로
실행하면 `content_classifier.py`, `case_store.py` 등 스크립트의 `python`
실행 자체가 "`python.exe` 프로그램을 실행하지 못했습니다. 지정한 로그온
세션이 없습니다"라는 Windows 오류로 실패했다. 처음에는 스크립트나
`requirements.txt` 설정 문제로 의심했지만, 같은 명령을
`--sandbox danger-full-access`로 재실행하자 `content_classifier.py`가
`preliminary_risk: {risk: High, ...}`를 정확히 반환하는 것을 확인했다.
즉 이 프로젝트 코드의 문제가 아니라, Codex CLI가 Windows에서 제한된
샌드박스 토큰으로 새 프로세스를 만들 로그온 세션을 확보하지 못하는
Codex 자체의 샌드박스 한계였다. README에 Windows에서는
`--sandbox danger-full-access` 또는 대화형 승인 모드로 실행해야 한다는
안내를 추가해 해결했다.

둘째, Codex가 PowerShell로 `law_search.py`, `심의_기준.md` 같은 UTF-8
한글 소스 파일을 읽을 때 텍스트가 깨져 보이는 현상을 발견했다.
`codex doctor`로 확인해보니 콘솔 코드페이지가 949(CP949)였다. 실제
디스크의 파일이 깨졌는지 `grep`으로 원본 바이트를 직접 확인한 결과
UTF-8로 정상 저장돼 있었고, 콘솔 표시 레이어에서만 깨지는 현상임을
확인했다. 다만 이 표시 문제가 Codex의 실제 판단(법령명 해석 등)에
영향을 주는지까지는 완전히 배제하지 못해 README의 한계로 남겼다.

셋째, 가장 중요한 발견인데, `LAW_API_KEY`를 실제로 발급받아
`law_search.py 자본시장법 금융소비자보호법`을 실행해보니 그동안 문서
전체(README, SKILL.md, 심의_기준.md, 이 답변초안)에서 근거로 삼던
"자본시장법 제49조(부당권유행위 금지)·제57조(투자광고)"가 **2020.3.24
삭제된 조문**이라는 사실을 API 응답("제49조 삭제 <2020.3.24>")으로
확인했다. AI가 작성한 초안 단계 요약이 삭제된 조문 번호를 그대로
가져다 쓴 것이었다. 두 조항이 금융소비자보호법 제정과 함께 각각
금융소비자보호법 제21조·제22조로 이관된 것을 법제처 API로 직접 조회해
확인하고, 관련 문서 전체를 정정했다. API 키 없이는 이 오류를 끝까지
발견하지 못했을 것이므로, "정보가 부족한 상황에서 위험도를 추정하지
않는다"는 SKILL.md의 원칙이 왜 필요한지를 스스로 증명한 사례이기도 하다.

**받아들이지 않은 AI 제안과 이유**
구체적으로 하나의 발화를 콕 집어 기억할 만한 순간은 없었다. 가장 가까운
사례는 위 "직접 판단한 부분"에 이미 적은 결정이다 — AI에게 맡겼다면
"원금보장"·"확정수익" 같은 표현도 문맥을 봐서 판단하는 순도 높은 LLM
기반 설계로 흘러가기 쉬웠을 텐데, 증권 광고 규제에서는 그 표현이 문맥과
무관하게 사실상 예외 없이 문제되므로 LLM 판단 이전에 규칙으로 먼저
걸러내는 구조를 택했다. 이걸 "AI 제안을 거부"한 사건이라기보다는, AI가
만든 초안의 기본값(유연한 LLM 판단)을 그대로 받아들이지 않고 안전장치를
앞단에 추가한 결정이라고 보는 게 더 정확하다.

## 5. 어떻게 검증했나요?

**원본 프로젝트 검증 이력 (재사용 근거, Ollama/ChromaDB 구버전 기준)**
- P0 스모크 테스트 4/4 통과: 법제처 API, ChromaDB, Ollama/EXAONE, 전체 흐름
- 콘텐츠 타입별 E2E 테스트 5종 통과(마케팅광고/영업제안서/약관/CS이메일문자/
  SNS앱푸시). 단, 이 검증은 로컬 LLM 기반 구버전 아키텍처 기준이라 이번
  Codex 플러그인 구조에는 직접 적용되지 않는다.

**입력 → 결과 예시 (스크립트 단위 검증 완료)**
"이 상품은 원금 보장에 확정 수익 연 5%를 제공합니다" (상품소개서) 입력 →
`content_classifier.py`가 `law_keywords: [자본시장법, 금융투자업규정,
금융소비자보호법]`, `preliminary_risk: {risk: High, reason: "투자상품을
예금처럼 오인하게 하는 표현은 자본시장법상 부당권유 소지가 매우 큼"}`을
정확히 반환하는 것을 확인.

**정상 상황 검증**
`case_store.py query`로 `data/cases.json`의 시드 8건 중 콘텐츠 타입이
일치하고 단어 겹침이 높은 사례가 유사도 순으로 반환되는 것을 확인.

**예외 상황 검증 (완료)**
- `LAW_API_KEY`를 설정하지 않고 `law_search.py`를 실행 → 위험도를 추정하지
  않고 즉시 `{"error": "LAW_API_KEY 환경변수가 설정되지 않았습니다..."}`를
  반환하며 종료 코드 1로 실패하는 것 확인.
- `case_store.py save`에 5자 미만 피드백("거절")을 입력 → `{"saved": false,
  "reason": "저품질 피드백 — 저장 거부"}` 반환 확인. 이후 위반 법령과 5자
  이상 피드백을 갖춘 정상 입력은 저장되고 `cases.json`에 반영되는 것도 확인.
- 법제처 API 실제 키로 `law_search.py 자본시장법 금융소비자보호법` 실행 →
  실제 조문 데이터가 정상 반환되는 것 확인. 그 과정에서 자본시장법
  제49조·제57조가 삭제된 조문이라는 사실을 발견해 문서 전체를 정정했다
  (위 4번 문항 "막혔던 지점" 참고).
- Codex CLI 플러그인 트리거 자체는 `LAW_API_KEY` 미설정 상태에서 이미
  검증 완료 — Codex가 조회 실패 상황에서 조문을 임의로 지어내지 않고
  "수동 법령 확인 필요"로 명시하는 것을 확인했다 (5~6단계 환각 방지
  원칙 확인).

**예외 상황 검증 (미완료)**
[ ] Codex가 **실제 조문 조회 결과가 있는 상태**에서 위반 가능 조항을
검색된 조문 안에서만 인용하는지, 조회되지 않은 법령을 인용했을 때
6단계 자체 검증에서 걸러내는지는 아직 Codex CLI로 End-to-End 실행하지
못했다 (지금까지는 API 키 없는 상태의 환각 방지 동작만 확인함).

**무엇을 의심했고 무엇이 아직 부족한가**
- 의심한 지점: 규칙 기반 선판정(원금보장 등 High 고정)과 Codex의 조문 기반
  최종 판단이 상충할 때 어느 쪽을 우선할지 — SKILL.md에 "근거 없이 사전
  판정보다 낮추지 않는다"는 지침으로 명시했으나, 이는 프롬프트 수준 강제라
  Codex가 지침을 어길 가능성 자체를 코드로 완전히 막지는 못한다.
- 또 다른 의심한 지점(결과적으로 맞았던 의심): AI가 초안에 적은 조문
  번호("자본시장법 제49조·제57조")를 그대로 믿지 않고 법제처 API로
  재검증했는데, 실제로 삭제된 조문이었다. 법령을 다루는 프로젝트에서는
  AI가 그럴듯하게 적은 조문 번호도 검증 전까지는 의심 대상으로 취급해야
  한다는 걸 확인했다.
- 아직 부족한 부분: 실제 조문 조회 결과를 놓고 Codex가 5~6단계(분석·인용
  검증)를 수행하는 전체 흐름은 아직 End-to-End로 재검증하지 못했다.

**테스트하며 고친 점**
- Ollama/ChromaDB/sentence-transformers 기반 원본 구조를 그대로 스크립트화
  하려다, 평가자가 무거운 로컬 모델을 설치해야만 실행되는 구조라는 문제를

  두 줄로 줄었고, 별도 설치 없이 스크립트가 바로 실행되는 것을 확인했다.
